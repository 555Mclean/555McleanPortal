Self-Updating Waiting Lists

How sign-ups flow from the website into the live parking and storage queues — automatically.

Until this is configured the portal still works: the sign-up forms fall back to opening a pre-filled email to the board, and the on-page queues simply show whatever is in data/waitlist.json. Setting it up adds automatic capture and a queue that refreshes itself.

1.How it works

Resident submits the on-site form → entry is recorded in a private Google Sheet (names & emails stay there) → a scheduled job reads the sheet every ~15 minutes → it rewrites data/waitlist.json with just the apartment numbers and order → the site redeploys with the updated queue.

Only apartment numbers ever appear publicly. Residents' names, emails, and phone numbers remain in the board's private responses sheet.

2.Create the Google Form

  1. At forms.google.com, create a form with these questions (short-answer / multiple-choice):
    • Which list? — options Parking Spot and Storage Unit
    • Apartment #
    • Full Name, Email, Phone (optional)
    • Spot Preference and First or second spot? (parking only, optional)
    • Optionally a Status column you fill in later — mark it assigned to drop someone from the public queue once they get a spot.
  2. In the form's Responses tab, click Link to Sheets to create the responses spreadsheet.

3.Publish the responses sheet as CSV

  1. Open the responses spreadsheet.
  2. File → Share → Publish to web.
  3. Choose the responses sheet and the Comma-separated values (.csv) format, then Publish.
  4. Copy the generated link — this is your WAITLIST_CSV_URL.

4.Tell the site about it

a) Enable auto-refresh of the queue. In the GitHub repository: Settings → Secrets and variables → Actions → New repository secret:

Name:  WAITLIST_CSV_URL
Value: <the published CSV link from step 3>

The Sync Waitlist workflow runs every 15 minutes (and on demand from the Actions tab). If your column headers differ from the defaults, add optional repository variables WAITLIST_COL_LIST, WAITLIST_COL_APT, WAITLIST_COL_REMOVE to point at them.

b) Send sign-ups straight into the form (optional but recommended). This lets residents use the site's own styled form instead of opening an email. In main.js, fill in WL_SUBMIT:

export const WL_SUBMIT = {
  url: 'https://docs.google.com/forms/d/e/FORM_ID/formResponse',
  fields: {
    list:       'entry.1111111111',
    name:       'entry.2222222222',
    unit:       'entry.3333333333',
    email:      'entry.4444444444',
    phone:      'entry.5555555555',
    preference: 'entry.6666666666',
    spot:       'entry.7777777777',
  },
};

To find each entry.… id: open the live form, right-click a field → Inspect, and read the name="entry.NNN" attribute (or use the form's “Get pre-filled link” feature). Leave url empty to keep the email fallback.

Same option for the Maintenance form. The Maintenance Request form supports the identical pattern via MAINT_SUBMIT in main.js — point it at its own Google Form to capture those submissions instead of opening an email. It defaults to the email flow when left blank.

5.That's it

  • New sign-ups appear in the public queue within ~15 minutes.
  • To remove someone (they got a spot, withdrew, etc.), set their Status cell to assigned in the sheet — or just delete the row.
  • You can still edit data/waitlist.json by hand; the next sync will reconcile it against the sheet.
